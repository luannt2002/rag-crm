## 2. AdapChunk — Mindset chunking adaptive

### 2.1 Vấn đề cần giải quyết

Hệ thống RAG phụ thuộc trực tiếp vào chất lượng phân đoạn (chunking). Fixed splitting (512 token mọi doc) gây:

- Chunk bị cắt giữa bảng/công thức → mất ngữ cảnh
- Vector embedding không phản ánh đúng ngữ nghĩa
- Retrieval trả về đoạn không liên quan
- LLM hallucinate vì thiếu thông tin nền

### 2.2 Kiến trúc 7 tầng (AdapChunk spec)

```
PDF → Mistral OCR → Markdown
  → Block Detection & Tagging (rule-based)
  → Feature Extraction → Document Profile
  → LLM Strategy Selector
  → Rule-based Cross-check
  → Chunking Executor (tôn trọng atomic block)
  → Narrate special blocks
  → Embedding → Qdrant + Metadata
```

### 2.3 Các loại block

| Tag | Atomic? | Ý nghĩa |
|-----|---------|---------|
| HEADING | Không | Cấu trúc phân cấp |
| TEXT | Không | Văn xuôi thường |
| TABLE | **Có** | Không được cắt đôi |
| FORMULA | **Có** | LaTeX giữ nguyên + context buffer |
| IMAGE | **Có** | Embed description từ OCR |

### 2.4 Bốn chiến lược chunking

| Strategy | Cơ chế | Phù hợp |
|----------|--------|---------|
| **HDT** | Phân đoạn theo heading hierarchy, structural_path | Luận văn, báo cáo có mục lục |
| **SEMANTIC** | Cosine similarity giữa câu liên tiếp, cắt tại điểm chuyển ngữ nghĩa | Sách giáo khoa, văn xuôi dài |
| **PROPOSITION** | LLM tách thành phát biểu nguyên tử, tự đủ nghĩa | Hợp đồng, quy chế, pháp lý |
| **HYBRID** | HDT macro + PROPOSITION micro (>300 từ) | Fallback an toàn |

### 2.5 Cross-check rules (lớp an toàn)

| Điều kiện | Hành động |
|-----------|-----------|
| confidence < 0.6 | Override → HYBRID |
| HDT nhưng heading < 5 | Override → SEMANTIC |
| SEMANTIC nhưng avg_text_block_length < 50 | Override → PROPOSITION |
| PROPOSITION nhưng avg_text > 300 và heading > 20 | Override → HDT |
| mixed_content_score > 0.4 và không chọn HYBRID | Cảnh báo (log) |

### 2.6 Narrate then Embed

Embedding model chỉ hiểu ngôn ngữ tự nhiên. Trước khi embed:

| Loại | Giải pháp |
|------|-----------|
| FORMULA LaTeX | LLM → câu mô tả; lưu LaTeX gốc trong metadata |
| TABLE Markdown | Linearize hàng hoặc LLM tóm tắt |
| IMAGE/GRAPH | Embed description từ OCR |
| TEXT | Embed trực tiếp |

**Metadata `original_content` quan trọng:** retrieval trả chunk → LLM đọc cả embedded text (ngữ nghĩa) lẫn original_content (số liệu chính xác).

### 2.7 Suy nghĩ Cursor về AdapChunk

> AdapChunk trả lời **một câu hỏi duy nhất**: "Tài liệu này nên được cắt như thế nào để retrieval sau này tìm đúng?"
>
> Ba nguyên tắc không thể bỏ: (1) Structure-first, (2) Atomic blocks, (3) Narrate then embed.
>
> Document Profile phải **rule-based, deterministic** — không dùng LLM — để làm ground truth cho cross-check.

## 3. Ragbot — Đánh giá thực tế

### 3.1 So sánh AdapChunk spec vs Ragbot implementation

| Thành phần | AdapChunk spec | Ragbot thực tế | Ghi chú |
|------------|----------------|----------------|---------|
| OCR | Mistral OCR (cloud) | Kreuzberg (self-hosted) | Data sovereignty |
| Strategy selector | LLM call | **Rule-based scorer** | 0 LLM cost/doc, deterministic |
| Vector DB | Qdrant | **pgvector** (PostgreSQL) | 1 DB, RLS-capable |
| Embedding | BGE-m3 / vietnamese-sbert | **jina-embeddings-v3** (1024-dim) | Single vendor embed+rerank |
| Block pipeline | Full block detection | **Flag ON, parser chưa emit** → no-op | ~60-70% AdapChunk |
| Narrate-then-embed | Required | Code có, **default OFF** | Chưa bootstrap-wired |
| Query graph | Không mô tả | **~21 LangGraph nodes** | Vượt xa AdapChunk scope |
| Multi-tenant | Không mô tả | **4-key identity + RLS** | Production concern |
| Anti-HALLU | Không mô tả | **Layered defense, HALLU=0 verified** | Sacred contract |

### 3.2 Ragbot đã LIVE

- Multi-tenant 4-key identity (app-filter solid)
- Async ingest 2-action (HTTP 202 + worker)
- 6 chunking strategies (`smart_chunk`)
- Rule-based selector + cross-check
- Parent/child small-to-big
- Hybrid retrieval (dense + BM25 + RRF)
- Rerank + CRAG grade + grounding judge
- Happy-case input control (template + normalizer + checker)
- HALLU=0 trên tested traps

### 3.3 Ragbot PARTIAL / gap

- Block pipeline (atomic protection) — code có, chưa wire
- Narrate-then-embed — code có, default OFF
- RLS DB enforcement — Phase 3 gated (DSN flip pending)
- Oversized sheet ingest — 224KB → 2643 chunks, có thể OOM
- OpenAI TPM rate-limit under load test

### 3.4 Suy nghĩ Cursor về Ragbot

> Ragbot build **thành công phần platform** (multi-bot, ingest async, query graph, anti-hallucination).
> Phần AdapChunk đầy đủ (block-aware + narrate) mới ~60–70%.
> README mô tả **đích đến production (Phase D–E)**, không phải **bước 1 (Gate 0)**.
> Cảm giác "app rác" = nhảy thẳng Phase D mà chưa pass Gate 0–1.

## 4. Tại sao RAG “chưa dùng được”?

### 4.1 Pattern thất bại phổ biến

| Triệu chứng | Nguyên nhân |
|-------------|-------------|
| Upload xong không trả lời được | Doc stuck `DRAFT`, embed fail im lặng |
| Trả lời sai / bịa | Chunk kém + chưa refuse khi no context |
| Setup 2 tiếng vẫn chưa chạy | Quá nhiều dependency trước khi "1 doc → 1 Q → 1 A" pass |
| Load test fail | Scale/optimize trước happy path |
| Happy-case format reject | Doc thật không match template |

### 4.2 Nguyên tắc vàng

```
Gate 0: Upload MD → chunk → embed → hỏi "X là gì?" → trả lời đúng (có citation)
Gate 1: Thêm PDF
Gate 2: Thêm refuse khi không có context
Gate 3: Thêm multi-bot isolation
Gate 4: Thêm async worker + 202
Gate 5: Thêm adaptive chunking (AdapChunk)
Gate 6: Thêm query graph phức tạp (rerank, CRAG, reflect)
```

**Không nhảy gate.** Mỗi gate = verify script + gold Q&A + pass/fail rõ.

### 4.3 Checklist "app dùng được"

```
□ docker compose up → API healthy < 30s
□ Upload 1 file MD → state=active trong < 10s
□ 3 câu hỏi gold → đúng fact, có citation
□ 2 câu trap (không có trong doc) → oos_template, KHÔNG bịa
□ Upload duplicate → dedup, không embed lại
□ Restart server → data còn, query vẫn đúng
□ Log mỗi request: chunks_retrieved, latency, token_used
```

## 5. Luồng INGEST chuẩn (U0–U7)

### 5.1 Sơ đồ tổng quan

```
Client upload
    │
    ▼
[U0] IDENTITY_VALIDATE ─── tenant + bot resolve
    │
    ▼
[U0.5] BOT_RESOLVE_4KEY ── BotConfig (chunk_size, model bindings...)
    │
    ▼
[U1] VALIDATE ──────────── size guard + dedup (content_hash)
    │
    ▼
[U2] PARSE ─────────────── format → structured text  ← MULTI-FORMAT GATE
    │
    ▼
[U3] CLEAN ─────────────── NFC, injection strip, normalizer
    │
    ▼
[U4] CHUNK ─────────────── AdapChunk / smart_chunk
    │
    ▼
[U5] ENRICH ────────────── contextual prefix (optional phase 2)
    │
    ▼
[U6] VN_SEGMENT ────────── underthesea tokenization (optional)
    │
    ▼
[U7] EMBED + STORE ─────── batch embed → pgvector → state=active
```

### 5.2 Upload pattern — 2 action (industry standard)

```
Action 1 (HTTP sync, <1s):
  POST /bots/{bot}/documents
  → validate size + dedup (content_hash)
  → fetch/accept raw content
  → INSERT documents (state='DRAFT', raw_content)
  → INSERT outbox event
  → HTTP 202 {document_id, state: "DRAFT"}

Action 2 (worker async):
  outbox → Redis Stream → consumer
  → đọc raw_content TỪ DB (không refetch URL!)
  → chạy pipeline U1–U7
  → flip state: DRAFT → enriching → embedding → active | failed
```

**Tại sao lưu raw_content vào DB?** Google Sheets `/edit?gid=` trả HTML login page nếu refetch sau.

### 5.3 Multi-format control (U2 + U3)

| Format | Parser | Output |
|--------|--------|--------|
| PDF | OCR (Kreuzberg/Mistral) | structured markdown |
| DOCX | python-docx / pandoc | markdown |
| XLSX/CSV | openpyxl | row-as-chunk hoặc table_csv |
| Google Sheets | export API | same as xlsx |
| HTML/Web | trafilatura | plain text / markdown |
| Plain MD | passthrough | happy-case checker |

**Happy-case mindset:** định nghĩa 1 template chuẩn (`## section` + `| table |`) + normalizer + checker score. Upload format lạ → normalizer cố → score thấp → warn/reject.

### 5.4 Input contract — 3 vòng phòng thủ

```
Vòng 1 — Gate ở upload (U1):
  MAX_SIZE, dedup hash, allowed MIME types

Vòng 2 — Normalize ở parse (U2–U3):
  Format → happy-case template
  Checker score → warn nếu < threshold

Vòng 3 — Quality ở chunk (U4):
  Strategy phù hợp structure
  Atomic block protection
  Cross-check override
```

### 5.5 Parent/child (small-to-big)

- **Child** (256 token): embed + retrieve — precision cao
- **Parent** (1024 token): không embed — expand lúc generate để LLM có context rộng

### 5.6 Schema DB tối thiểu

```sql
documents (
  id, record_bot_id, filename, content_hash,
  raw_content, state, error_message, created_at
)

document_chunks (
  id, document_id, record_bot_id,
  chunk_index, content,              -- text embed
  embedding vector(1024),
  content_tsv tsvector,              -- BM25
  parent_id, structural_path,
  strategy_used, block_types,
  original_content, metadata jsonb
)
```

### 5.7 Verify scripts (bắt buộc mỗi gate)

```bash
# Gate ingest
python scripts/verify_ingest.py \
  --file samples/spa_prices.md \
  --bot-id demo \
  --expect-chunks-min 3 \
  --expect-active

# Gate retrieval
python scripts/verify_retrieval.py \
  --bot-id demo \
  --query "giá massage bao nhiêu" \
  --expect-chunk-contains "300.000"
```

## 6. Luồng QUERY chuẩn (Q1–Q7)

### 6.1 MVP 5 bước (đủ dùng)

```
Q1 guard_input        → block prompt injection
Q2 embed_query        → cùng model ingest
Q3 retrieve           → hybrid dense + BM25, filter bot_id, top_k=10
Q4 rerank (phase 2)   → top 5
Q5 generate           → temp=0, chỉ context, refuse nếu rỗng
Q6 guard_output       → grounding check (phase 2)
```

### 6.2 Ragbot full query graph (~21 nodes)

```
guard_input → check_cache + understand (parallel)
   ├─(cache hit)──────────────────────────────────────────► persist
   └─► condense_question / router / query_complexity
        → rewrite | decompose | adaptive_decompose | speculative_retrieve
        → retrieve / graph_retrieve
        → rrf_round_robin → rerank → mmr_dedup → neighbor_expand(opt)
        → grade (CRAG 3-state) → rewrite_retry (loop)
        → generate → critique_parse(opt) → guard_output
             → reflect → persist → END
```

### 6.3 Generate contract — chống bịa

```
System: {bot.system_prompt}   ← bot owner cung cấp, app KHÔNG sửa

User:
<context>
{retrieved chunks với [chunk_id] header}
</context>

Question: {user_question}

Rules:
- Chỉ trả lời từ <context>
- Nếu không đủ thông tin: trả lời EXACT oos_template
- Trích dẫn [chunk_id] khi có fact cụ thể
```

**temperature = 0.0** luôn.

**Refuse short-circuit (bắt buộc MVP):**

```python
if not chunks:
    return bot.oos_answer_template  # KHÔNG gọi LLM
```

### 6.4 Hybrid retrieval

```python
dense = vector_search(query_emb, bot_id, k=20)
sparse = bm25_search(query_text, bot_id, k=20)
merged = rrf_fuse(dense, sparse, k=10)
```

VN text: BM25 + dense tốt hơn pure vector.

### 6.5 Verify query end-to-end

```bash
python scripts/verify_query_flow.py \
  --bot-id demo \
  --question "Giá gói massage 60 phút?" \
  --expect-answer-contains "300" \
  --expect-no-hallucination
```

Gold set tối thiểu: **11 câu / 1 bot / 3 lượt stable**.

### 6.6 Latency target (Ragbot measured)

| Tier | Intent | LLM calls | p50 |
|------|--------|-----------|-----|
| T1 | Greeting/chitchat | 1 | <1s |
| T2 | Factoid/FAQ RAG | 2–3 | 5–7s |
| T3 | Multi-hop/aggregation | 5–8 | 9–11s p95 |

## 7. Khung DEBUG RAG — 8 Step × 3 Cụm

### 7.1 Bảng tổng hợp

| Cụm | Step | Verify gì (DATA) | Metric |
|-----|------|------------------|--------|
| **INGEST** | 1. PARSE | file → markdown CÓ CẤU TRÚC? heading/table/atomic giữ? | BI |
| **INGEST** | 2. CHUNK | strategy đúng? atomic không cắt? parent-child? structural_path? | SC, ICC, DCC |
| **INGEST** | 3. EMBED | null_non_parent=0? dim 1024? narrate (nếu on)? | — |
| **INGEST** | 4. STORE | tsvector + embedding search được? stats-index? | RC (index health) |
| **QUERY** | 5. RETRIEVE | answer chunk vào top-K? Recall: dense+BM25+RRF+rerank | RC, Context Precision |
| **QUERY** | 6. GENERATE | LLM trả từ context? không false-refuse? không bịa? | Answer Correctness, Faithfulness |
| **QUERY** | 7. GUARD/REFLECT | grounding check? refusal đúng? | HALLU rate |
| **EVAL** | 8. SCORE | load-test → COVERAGE + HALLU + layer-attribution + intrinsic/retrieval/answer | All metrics |

### 7.2 Nguyên tắc debug

```
Mỗi step fail → trace NGƯỢC theo request_id / document_id / trace_id
  → xem step nào fail
  → model nào được gọi
  → token nào tiêu tốn
  → latency bao nhiêu
```

### 7.3 Layer attribution (Step 8)

Khi answer fail, biết fail ở layer nào:

```
fail_answer
  → retrieve_miss?     (Step 5 — RC thấp)
    → chunk_bad?       (Step 2 — SC/ICC/DCC/BI)
      → parse_bad?     (Step 1 — BI)
        → wrong_strategy? (Step 2 — strategy log)
  → generate_fail?     (Step 6)
  → guard_block?       (Step 7 — false refuse)
```

### 7.4 Scope kiểm thử (từ checklist team)

| Scope | Mô tả |
|-------|-------|
| Ingest / Upload đa-format | PARSE step — mọi MIME type |
| Chunking / AdapChunk L1–L7 | CHUNK + EMBED + STORE |
| Answer / Generation (sacred-10) | GENERATE — 10 câu sacred không được sai |
| Chat-entry + Test-chat (2 scope) | Entry point khác nhau, cùng pipeline |
| Retrieval (hybrid/RRF/rerank/CRAG) | RETRIEVE step |
| Multi-tenant / RLS / leak | Identity isolation |
| Cost-Log Center (token_ledger/CRM) | Ledger + rollup |
| Domain-neutral / hardcode sweep | 0 brand literal trong src |
| Multi-language | per-bot language config |
| Cost / Perf / Latency | 5 tiêu chí |

## 8. 5 tiêu chí Expert RAG

```
         Accuracy/Faithfulness = 100%
              ▲
              │    ★ Expert RAG sweet spot
              │   ╱  (hybrid + guard + cache)
              │  ╱
    Cost ◄────┼────► Latency
              │  ╲
              │   ╲ UX (refuse rõ, citation, streaming)
              ▼
         Performance (throughput, concurrent)
```

### 8.1 Nhanh (Latency thấp)

| Technique | Tiết kiệm |
|-----------|-----------|
| Cache L1 Redis exact hash | ~100% repeat Q |
| Cache L2 semantic @0.97 | ~80% FAQ |
| Adaptive router — skip rerank cho factoid | −40% latency |
| Multi-query speculative (parallel) | −21% cost, −779ms |
| Adaptive context prune | −18% cost, −619ms |
| Parent expand-only (không embed parent) | −50% embed cost/doc |

### 8.2 Đúng/Chuẩn (Faithfulness → 100%)

| Layer | Defense |
|-------|---------|
| Empty context | refuse → oos_template |
| CRAG grade | irrelevant → retry/refuse |
| Cliff rerank floor | drop weak chunks |
| Citation validation | chunk_id phải trong retrieved set |
| Grounding judge | SUPPORTED/NOT |
| Self-RAG reflect (opt) | completeness check |

**Metric:** HALLU=0 trên trap set + Faithfulness RAGAS + Answer Correctness gold set.

### 8.3 UX cao

- Streaming response (SSE)
- Refuse message rõ (`bots.oos_answer_template` per bot)
- Citation clickable → link doc source
- Upload progress (poll document.state)
- Multi-language (detect query lang → response lang match)
- Error transparent (failed ingest → error_message cụ thể)

### 8.4 Performance cao

- pgvector HNSW + tsvector GIN
- Partition `request_trace` by month
- CRM đọc `stats_*_daily`, không scan ledger
- Batch embed, async ingest worker
- Horizontal query (stateless API) + vertical ingest worker

### 8.5 Cost thấp

- Rule-based strategy selector (0 LLM/doc)
- temp=0, deterministic transforms
- Per-bot cost cap (`plan_limits.monthly_token_cap`)
- Cascade router → cheap model cho simple intent
- Cache bust chỉ khi `corpus_version` đổi
- Rollup stats offline

## 9. Identity hierarchy — Multi-tenant

### 9.1 Cây phân cấp

```
Platform (admin hệ thống)
  └── Tenant (record_tenant_id)          ← billing cap, CRM org
        └── Workspace (workspace_id)     ← team/project slug
              └── Bot (bot_id)             ← 1 KB, 1 system_prompt, 1 ngôn ngữ
                    └── Channel (channel_type)  ← web | zalo | api | messenger
```

### 9.2 4-key contract

| Layer | Key public | Key internal |
|-------|-----------|--------------|
| Tenant | JWT `record_tenant_id` | `tenants.id` |
| Workspace | `workspace_id` slug | `workspaces.id` |
| Bot | `bot_id` slug | `bots.record_bot_id` |
| Channel | `channel_type` | part of 4-key |

**Unique:** `(record_tenant_id, workspace_id, bot_id, channel_type)` → 1 bot instance.

**Sau resolve 1 lần ở boundary → mọi query dùng `record_bot_id` + app-filter.**

### 9.3 Multi-language

- Không tách bot theo ngôn ngữ
- Mỗi bot: `primary_language` + `supported_languages[]`
- Embed/rerank model bind per-bot
- Query log ghi `detected_query_language`

### 9.4 RLS isolation

```
Request path:  SET LOCAL app.tenant_id → NOBYPASSRLS role (ragbot_app)
Worker path:   BYPASSRLS role (ragbot_system) — cross-tenant rollup only
Log query:     tenant admin → WHERE record_tenant_id = ctx.tenant
               platform admin → stats_platform_daily (system role)
```

pipeline_step_log  → N rows / request (mỗi step U/Q)

### 10.5 Aggregation tables (rollup worker hourly/daily)

```sql
stats_bot_daily       (record_bot_id, stat_date)     -- drill-down bot
stats_workspace_daily (record_tenant_id, workspace_id, stat_date)
stats_tenant_daily    (record_tenant_id, stat_date)
stats_platform_daily  (stat_date)                     -- admin hệ thống
```

Mỗi bảng rollup chứa: requests, tokens, cost, refusal_count, hallucination_count, cache_hit_count, latency_p50/p95, tokens_by_purpose JSONB, tokens_by_model JSONB.

### 10.6 Luồng ghi log

```
Client → API boundary
  → INSERT request_trace (started_at, identity)
  → run pipeline (request_id injected)

  loop each step:
    → INSERT pipeline_step_log (started)
    → execute step
    → if external API:
        → TokenLedgerService.record() → INSERT token_ledger
    → UPDATE pipeline_step_log (ended, verify)

  → UPDATE request_trace (ended, aggregates, status)
  → response to client

  [hourly worker]
  → aggregate token_ledger → stats_*_daily
```

### 10.7 Quy tắc sacred

| Rule | Lý do |
|------|-------|
| Mọi external call qua `TokenLedgerService.record()` | Không gọi OpenAI/Jina trực tiếp |
| `request_id` inject từ boundary | Trace xuyên pipeline |
| Ghi ledger trong cùng transaction (hoặc outbox) | Không mất log khi crash |
| `idempotency_key` = `{request_id}:{purpose}:{seq}` | Retry không double-count |
| Aggregate vào request_trace cuối request | Dashboard đọc 1 row |
| Rollup worker hourly | CRM query stats_*_daily, không scan ledger |

### 10.8 LLMGateway — wrapper bắt buộc

```python
class LLMGateway:
    """Mọi LLM call đi qua đây — auto-log token."""
    async def complete(self, purpose: str, **kwargs) -> LLMResponse:
        started = utcnow()
        resp = await self._client.complete(**kwargs)
        await self._ledger.record(
            request_id=self._ctx.request_id,
            provider=resp.provider,
            model=resp.model,
            purpose=purpose,
            input_tokens=resp.usage.input,
            output_tokens=resp.usage.output,
            started_at=started,
            ended_at=utcnow(),
        )
        return resp
```

**Không có path nào gọi `openai.chat.completions.create()` trực tiếp.**

```sql
-- Tenant: token 7 ngày
SELECT stat_date, total_tokens, total_cost_usd, chat_requests
FROM stats_tenant_daily
WHERE record_tenant_id = $1 AND stat_date BETWEEN $2 AND $3;

-- Drill-down: bot tốn nhất
SELECT b.bot_id, s.total_tokens, s.total_cost_usd
FROM stats_bot_daily s JOIN bots b ON b.record_bot_id = s.record_bot_id
WHERE s.record_tenant_id = $1 AND s.workspace_id = $2
ORDER BY s.total_cost_usd DESC;

-- Trade log chi tiết 1 request
SELECT rt.*, json_agg(psl) AS steps, json_agg(tl) AS token_calls
FROM request_trace rt
LEFT JOIN pipeline_step_log psl ON psl.request_id = rt.request_id
LEFT JOIN token_ledger tl ON tl.request_id = rt.request_id
WHERE rt.request_id = $1 GROUP BY rt.id;
```

## 12. Evaluation metrics (Table 5)

### 12.1 Intrinsic metrics (chunk quality)

| Metric | Ý nghĩa | Step |
|--------|---------|------|
| **SC** Size Compliance | Chunk đúng size policy | Step 2 CHUNK |
| **ICC** Intrachunk Cohesion | Nội dung trong chunk liên quan | Step 2 |
| **DCC** Contextual Coherence | Context buffer atomic block đủ | Step 2 |
| **BI** Block Integrity | TABLE/FORMULA/IMAGE không cắt | Step 1–2 |
| **RC** Retrieval Completeness | Chunk retrieved chứa đủ fact | Step 5 |

### 12.2 Extrinsic metrics (end-to-end)

| Metric | Ý nghĩa | Tầm quan trọng |
|--------|---------|----------------|
| Faithfulness | Độ trung thực vs nguồn | Cao — hallucination |
| Context Precision | Chunk retrieved đúng info? | Cao — retrieval quality |
| Context Recall | Info cần thiết retrieve hết? | Cao — coverage |
| Answer Correctness | Trả lời đúng câu hỏi? | Cao |
| Answer Relevance | Câu trả lời relevant? | Trung bình |
| Strategy Selection Accuracy | Chọn đúng strategy? | Cao — AdapChunk mới |
| HALLU rate | Tỉ lệ bịa | **Sacred = 0** |

### 12.3 Gold question design

| Loại câu hỏi | Mục đích | Ví dụ |
|--------------|----------|-------|
| Nằm gọn 1 đoạn | Baseline | "CNN là gì?" |
| Cần context heading | Chunk mất heading → sai | "Phương pháp nào trong chương 3?" |
| Liên quan bảng | Bảng cắt đôi → mất data | "Cảm biến nào sai số thấp nhất?" |
| Liên quan công thức | Formula tách khỏi giải thích | "F1-score tính như thế nào?" |
| Tổng hợp nhiều đoạn | Multi-chunk retrieval | "Tóm tắt kết quả thực nghiệm" |
| Tham chiếu chéo | Mối quan hệ giữa phần | "Đề xuất chương 3 khác gì chương 2?" |
| **Trap (không có trong doc)** | **HALLU test** | "Thời tiết HN hôm nay?" |

Mỗi doc: **15–20 câu hỏi** + đáp án gold + đoạn văn gốc.

### 12.4 Ablation configs

| Config | Mô tả |
|--------|-------|
| Baseline | Fixed recursive 512 token |
| HDT-only | Luôn HDT |
| SEMANTIC-only | Luôn SEMANTIC |
| PROPOSITION-only | Luôn PROPOSITION |
| AdapChunk | Full adaptive |
| AdapChunk (no cross-check) | Không rule override |

**Wilcoxon p < 0.05:** so sánh AdapChunk vs baseline trên RC — chứng minh cải thiện có ý nghĩa thống kê.

## 13. Trade-off & quyết định kiến trúc

### 13.1 Giữ từ Ragbot / AdapChunk

| Giữ | Lý do |
|-----|-------|
| 2 pipeline ingest/query tách biệt | Scale độc lập |
| temp=0 + refuse khi no context | HALLU=0 sacred |

| Parent/child chunks | Precision + context rộng |

| Verify scripts + gold Q&A | Đo được "dùng được" |

Để phân tích chuyên sâu và xác định "tầng" (layer) gây ra bug dựa trên dữ liệu bạn đã có, chúng ta cần chia hệ thống thành **4 tầng logic (Stack Analysis)**. Khi một câu hỏi bị fail, chúng ta sẽ soi nó qua từng tầng này theo thứ tự từ dưới lên trên.

Dưới đây là khung phân tích tầng để bạn truy vấn sâu vào từng lỗi:

### 1. Tầng Data/Retrieval (DSI - Document Search Index)

* **Vấn đề**: Tài liệu có đáp án nhưng không được kéo về (retrieval miss) hoặc dữ liệu bị "bóp méo" trước khi vào LLM.
* **Dấu hiệu nhận biết**:
* Tài liệu gốc (raw chunks) chứa thông tin nhưng `chunks` trả về bằng 0 hoặc không chứa từ khóa quan trọng.
* Thông tin bị đè bởi `stats_lookup` (như các lỗi **BRAND** bạn đã liệt kê).


* **Cách truy vấn chuyên sâu**:
* **Trace DSI**: Lấy ID của câu query, truy vấn lại `semantic_search_results` cho câu đó. Kiểm tra xem `top_k` có chứa chunk mang đáp án không?
* **Stats-Weighting Check**: Kiểm tra các file định nghĩa `synthetic_chunk`. Tại sao `score=1.0` lại được gắn cho một thống kê trống/không có giá trị? (Đây chính là tầng gây ra lỗi **BRAND**).



### 2. Tầng Coreference/Context (State Management)

* **Vấn đề**: Hệ thống không duy trì được "trạng thái" của hội thoại (conversation history).
* **Dấu hiệu nhận biết**:
* Các lỗi **COREF** (như lỗi S-058, B-060).
* Hệ thống "quên" đối tượng (referent) ở câu trước dù `turns=2`.


* **Cách truy vấn chuyên sâu**:
* **Session State Dump**: Truy vấn log `conversation_history` của `connect_id` đó. Xem hệ thống có thực sự lưu lại tên SKU/dịch vụ của `turn-1` không, hay nó đã bị xóa/overwrite bởi `turn-2`.
* **Referent Resolution Trace**: Kiểm tra `prompt` cuối cùng gửi đến LLM. Nếu trong prompt không xuất hiện context của `turn-1`, thì bug nằm ở tầng **Middleware/Context Injection** (chưa ghép nối được history).



### 3. Tầng Reasoning/Prompt (LLM Logic)

* **Vấn đề**: Thông tin đã vào prompt (đầy đủ context + dữ liệu) nhưng LLM xử lý sai (bịa thông tin, lười tính toán).
* **Dấu hiệu nhận biết**:
* Các lỗi **WK (World Knowledge Fabrication)**: LLM có chunk về "Triệt lông", nhưng tự bịa thêm "Hàn Quốc".
* Lỗi **COVER (Comparison/Aggregation)**: LLM nhận đủ dữ liệu nhưng không biết so sánh hoặc tổng hợp.


* **Cách truy vấn chuyên sâu**:
* **Prompt-Only Testing**: Copy toàn bộ `prompt` (cả context + query) vào một môi trường test riêng (playground).
* **Temperature & System Prompt Analysis**: Nếu cùng prompt đó mà 5 lần chạy ra 5 đáp án khác nhau, thì bug nằm ở `temperature` hoặc `System Prompt` thiếu tính ràng buộc.
* *Giả thuyết*: Các lỗi WK của bạn nằm ở tầng này — cần thêm "Anti-hallucination instruction" vào System Prompt.



### 4. Tầng Output/Formatting (Harness & Grading)

* **Vấn đề**: Bot trả lời đúng nhưng bị "gắn nhãn sai" bởi eval-agent.
* **Dấu hiệu nhận biết**:
* Các lỗi **NOTFAIL** (như S-019, G-068).
* Bot trả lời đúng ý nhưng format không khớp mong đợi.


* **Cách truy vấn chuyên sâu**:
* **Verdict Audit**: Kiểm tra file `eval_verdict.json` cho ID đó. Xem tại sao Agent lại chấm `WRONG` trong khi nội dung câu trả lời khớp thực tế.
* **Threshold Audit**: Nếu `chunks=0`, hệ thống tự động gán `WRONG` mà không đọc câu trả lời. Đây là bug của **Harness Layer**, không phải của Bot.

Chúng ta sẽ sử dụng chiến thuật **"Trace-to-Log"** (Truy vết đến log). Bạn hãy thực hiện các truy vấn/phân tích theo trình tự này trên bộ dữ liệu fail-stable:

### 1. Phẫu thuật tầng Retrieval (Tập trung lỗi BRAND)

Lỗi BRAND của bạn (ví dụ: `B-010`, `B-011`, `S-037`) cho thấy hệ thống đang bị "ngộ độc" bởi Synthetic Chunk. Hãy dump log của 10 câu BRAND này và trả lời 3 câu hỏi sau:

* **Truy vấn 1 (Cấu trúc đầu vào):** Trong log `raw_prompt` gửi cho LLM, hãy kiểm tra xem `context_chunks` có bao nhiêu chunk?
* *Nghi vấn:* Có phải `synthetic_chunk` (với `score=1.0`) luôn nằm ở vị trí đầu tiên (được ưu tiên cao nhất) và đẩy `raw_chunk` (chứa dữ liệu thật) xuống dưới hoặc ra ngoài window không?


* **Truy vấn 2 (Score Inflation):** Tại sao `synthetic_chunk` lại được hệ thống gán `score=1.0` (mặc định)?
* *Mổ xẻ:* Tìm trong source code đoạn `_do_stats_lookup`. Có phải nó đang tạo ra một chunk giả định với metadata `price: null` hoặc `status: none` và áp đặt score tuyệt đối khiến thuật toán RAG bỏ qua các chunk thực tế?


* **Truy vấn 3 (Semantic Filter):** Kiểm tra `semantic_search_results` cho câu hỏi "Lốp Rovelo 195/70R14 giá bao nhiêu?".
* *Mổ xẻ:* Nếu kết quả search trả về chính xác dòng lốp đó, nhưng prompt gửi đi lại không có, thì **Middleware** đang lọc bỏ chunk đó dựa trên filter nào?



### 2. Phẫu thuật tầng Reasoning (Tập trung lỗi WK - Fabrication)

Lỗi này (`S-044`, `S-045`) xảy ra khi dữ liệu thiếu, nhưng LLM vẫn "tự tin" bịa ra thông tin. Đây là lỗi tại **tầng LLM Alignment**.

* **Truy vấn 1 (Grounding Check):** Kiểm tra `System Prompt` đang dùng.
* *Câu hỏi:* Có câu lệnh "If the information is not provided in the context, state that you do not know" hay không? Nếu có, tại sao nó bị vô hiệu hóa?


* **Truy vấn 2 (Attention Drift):** So sánh `raw_prompt` của một câu trả lời đúng (CORRECT) và một câu WK (WRONG).
* *Nghi vấn:* Có phải trong các câu WK, LLM đang bị đánh lạc hướng bởi các chi tiết có vẻ "chuyên gia" trong System Prompt (ví dụ: mô tả về da, SMAS) khiến nó cố gắng hoàn thiện câu trả lời thay vì dựa hoàn toàn vào Context?



### 3. Phẫu thuật tầng State/Context (Tập trung lỗi COREF)

Lỗi này (`S-058`, `B-060`) là lỗi "mất kết nối".

* **Truy vấn 1 (Session State Dump):** Truy vấn `last_turn_context`.
* *Mổ xẻ:* Hãy xem `connect_id` đó. Trong `turn-1`, hệ thống đã ghi nhận entity là "Ultherapy" hay "195/65R15" vào `slots` chưa?


* **Truy vấn 2 (Query Expansion):** Khi người dùng hỏi "Quy trình thực hiện dịch vụ đó", hệ thống có thực hiện bước **"Query Expansion"** (nội suy từ "đó" thành tên dịch vụ gốc) trước khi gọi DSI không?
* *Nghi vấn:* Nếu không expand, thì DSI sẽ tìm kiếm từ khóa "đó" — đương nhiên kết quả sẽ sai lệch hoàn toàn.
